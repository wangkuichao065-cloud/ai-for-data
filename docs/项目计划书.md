# 数据分析系统综合实践 — 项目计划书

---

## 一、项目概述

### 1.1 项目名称

基于知识图谱与大模型的课程数据分析平台

### 1.2 项目背景

本系统为「数据分析系统综合实践」课程大作业，旨在综合运用 Neo4j 知识图谱、大语言模型 RAG 检索增强生成、Stable Diffusion 数字人等前沿技术，构建一个面向机器学习与数据挖掘两门课程的知识分析平台。系统通过知识图谱对课程知识点进行结构化建模，结合 RAG 技术实现智能问答，并利用数字教师提供沉浸式教学交互体验。

### 1.3 项目目标

将机器学习与数据挖掘两门课程的知识体系构建为 Neo4j 知识图谱，支持可视化展示与路径查询。基于 DeepSeek 大语言模型与 FAISS 向量检索构建 RAG 管线，实现课程知识的智能问答，回答可追溯引用来源。利用 Stable Diffusion 生成数字教师形象，结合 TTS 实现文字+语音的多模态教学交互。提供数据分析仪表盘，可视化展示提问趋势、知识点热度、用户活跃度和学习掌握度等维度数据。

### 1.4 技术栈总览

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端 | Vue 3 + Element Plus + ECharts + neovis.js | CDN 单 HTML 文件方案，无需构建 |
| 后端 | Python + FastAPI | 自动生成接口文档，异步高性能 |
| 图数据库 | Neo4j 5.x | 知识图谱建模与查询 |
| 关系数据库 | MySQL 8.0 | 用户、会话、文件等业务数据 |
| 向量库 | FAISS | 文档分块向量索引与相似度检索 |
| 大语言模型 | DeepSeek-R1-Distill-Qwen-7B | Ollama 本地部署，Q4 量化 |
| Embedding | BAAI/bge-small-zh-v1.5 | 中文文本向量化，512 维 |
| RAG 框架 | LangChain | 编排检索-增强-生成管线 |
| 数字人 | Stable Diffusion v1.5 + TTS | 数字教师形象生成与语音 |
| 部署 | Docker（可选） | 容器化部署 |

### 1.5 硬件环境

| 项目 | 配置 |
|------|------|
| GPU | 8GB 显存（影响大模型部署策略） |
| CPU | — |
| 内存 | 16GB+ |
| 存储 | 50GB+（模型+数据+日志） |

8GB 显存策略：DeepSeek 7B Q4 量化（约 4.5GB）与 Stable Diffusion 1.5（约 3.5GB）交替加载，不同时满载运行。系统提供模型切换 API，在问答时加载 LLM，在生成数字教师图片时切换为 SD。

## 二、系统架构设计

### 2.1 整体架构

系统采用前后端分离的三层架构：前端通过 HTTP API 与 FastAPI 后端通信，后端统一协调 Neo4j 图数据库、MySQL 关系数据库和 FAISS 向量库。大语言模型通过 Ollama 本地部署，由后端 LangChain 管线编排调用。Stable Diffusion 作为可选服务，在需要生成数字教师形象时按需加载。

### 2.2 RAG 双路检索管线

当用户提问时，系统同时执行两条检索路径。第一条是向量检索路径：问题经 Embedding 模型向量化后在 FAISS 中搜索相似文档分块，返回 Top-K 相关内容。第二条是图谱检索路径：从问题中提取关键词，在 Neo4j 中查询相关知识点及其上下文（前置依赖、相似知识点、关联算法）。两路结果合并后拼入 Prompt，送入 DeepSeek 生成最终回答。这种设计确保回答既有文档级支撑，又有知识结构级增强。

### 2.3 前端页面规划

首页（仪表盘）展示系统概览，包含 KPI 卡片、提问趋势折线图、热门知识点进度条、课程知识图谱径向图等数据可视化组件。知识图谱页面使用 neovis.js 或 ECharts Graph 渲染交互式知识图谱，支持节点点击查看详情、关系高亮、课程筛选。智能问答页面提供流式问答界面，支持 Markdown 渲染回答、引用来源展示、多轮对话和会话历史管理。数字教师页面展示数字人形象，支持文字交互、语音播报、表情切换。数据分析页面提供多维度数据可视化，包括提问趋势、知识点热度图、用户活跃度和满意度分析。

## 三、模块任务分解

### 3.1 知识图谱模块

该模块负责课程知识点的 Neo4j 图谱建模。需要从机器学习和数据挖掘两门课程中梳理知识点体系，定义 5 类节点（课程、章节、知识点、算法、应用场景）和 6 类关系（包含、依赖、应用于、相似于、实现、属于），编写 Cypher 初始化脚本完成数据导入。后端实现图谱可视化数据接口、节点详情查询、关键词搜索、学习路径查询和统计接口。前端使用 neovis.js 渲染力导向图，支持缩放、拖拽、节点点击和高亮。

### 3.2 智能问答模块

该模块是系统的核心交互功能。需要搭建 LangChain + FAISS + DeepSeek 的完整 RAG 管线。文档处理流程包括：上传 PDF/TXT/DOCX → 文本分块（chunk_size=500, overlap=50）→ bge-small-zh 向量化 → FAISS 入库。问答流程包括：用户提问 → 双路检索（FAISS+Neo4j）→ Prompt 拼接 → DeepSeek 流式生成 → 引用来源标注。支持多轮对话（基于 session_id 维护上下文）、回答反馈评分、会话历史管理。前端实现 SSE 流式渲染，支持 Markdown 格式显示和代码高亮。

### 3.3 数字教师模块

该模块提供沉浸式教学体验。数字教师形象方案分两个层次：基础方案使用 Stable Diffusion 生成静态教师形象图片，配合 CSS 动画实现简单表情切换；进阶方案使用 Live2D 模型实现动态人物，嘴巴动作与语音同步。TTS 语音合成方案使用 edge-tts 或 CosyVoice，将数字教师的文字回答转为语音播放。后端实现教师对话接口（复用 RAG 管线，附加情绪判断和动作指令）、形象生成接口和 TTS 接口。由于 8GB 显存限制，SD 模型在需要生成图片时加载，日常问答仅使用 LLM。

### 3.4 数据分析模块

该模块提供系统级数据可视化。后端聚合各模块数据，提供仪表盘概览、提问趋势分析、知识点热度、用户活跃度、满意度分析和知识掌握度雷达图等接口。前端使用 ECharts 绘制折线图、柱状图、热力图、雷达图等多种图表，配色与仪表盘整体紫蓝渐变风格统一。

### 3.5 用户认证模块

实现用户注册、登录、JWT Token 认证、角色权限控制（student/teacher/admin）。使用 bcrypt 哈希存储密码，FastAPI Depends 注入当前用户。学生角色可提问和查看图谱，教师角色可上传知识库文档和管理知识点，管理员角色可管理系统配置和切换模型。

### 3.6 文件管理模块

支持上传 PDF/TXT/DOCX/MD 文档到 RAG 知识库，自动分块向量化。提供文件列表、处理状态查询、删除文件（同时清理 FAISS 向量和文件系统）、重建向量索引等功能。

### 3.7 系统管理模块

提供系统健康检查（Neo4j/Ollama/FAISS/MySQL 状态监控）、大模型状态查询、GPU 显存使用情况、模型切换（LLM 与 SD 交替加载）、系统配置管理和操作日志记录。

## 五、开发阶段与时间安排

### 5.1 阶段总览

项目分为六个阶段，总计约 5-6 周。

第一阶段：环境搭建与基础架构（第 1 周）。完成开发环境搭建，包括 Python/Anaconda、Neo4j、MySQL、Ollama+DeepSeek 模型、FAISS 依赖安装。初始化项目目录结构，搭建 FastAPI 基础框架，配置 CORS 和统一响应格式。搭建前端 HTML 骨架，配置 Vue3+ElementPlus+ECharts CDN 引入。

第二阶段：知识图谱构建（第 2 周）。梳理机器学习和数据挖掘两门课程的知识体系，设计节点和关系类型。编写 Neo4j 初始化 Cypher 脚本，完成约 150+ 知识节点的导入。实现后端图谱 API（可视化数据、节点详情、搜索、路径查询、统计）。前端使用 neovis.js 渲染交互式知识图谱。

第三阶段：RAG 问答核心（第 3 周）。搭建 LangChain RAG 管线，实现文档上传→分块→Embedding→FAISS 入库全流程。实现双路检索（FAISS+Neo4j）和 Prompt 拼接策略。对接 Ollama DeepSeek 模型，实现流式和非流式问答。前端实现 SSE 流式渲染、Markdown 显示、引用来源展示和多轮对话管理。

第四阶段：数字教师与文件管理（第 4 周）。部署 Stable Diffusion v1.5，实现数字教师形象生成。集成 TTS 语音合成，实现文字转语音。前端搭建数字教师交互界面，实现形象展示、文字聊天和语音播放。完成文件上传管理模块，支持 PDF/TXT/DOCX 上传和向量化。

第五阶段：数据分析与系统管理（第 5 周）。实现数据分析仪表盘全部接口和前端图表可视化。完成用户认证模块（注册、登录、JWT、角色权限）。实现系统管理模块（健康检查、模型状态、配置管理、操作日志）。完成各模块联调测试。

第六阶段：优化与答辩准备（第 6 周）。全面功能测试和 Bug 修复。前端界面美化和交互优化。性能优化（RAG 检索速度、模型加载时间）。编写答辩 PPT 和演示脚本，准备答辩演示。

## 六、风险评估与应对

### 6.1 技术风险

8GB 显存不足是最大硬件风险。DeepSeek 7B Q4 量化需要约 4.5GB 显存，Stable Diffusion 1.5 需要约 3.5GB，两者同时加载会超出显存。应对方案是实现模型交替加载机制，问答时加载 LLM 卸载 SD，生成图片时反向操作。系统提供模型切换 API 供管理员手动控制。如果 7B 模型仍然不稳定，可降级为 DeepSeek-R1-Distill-1.5B（仅需约 1.5GB 显存），牺牲部分回答质量换取稳定性。

RAG 检索质量风险。如果 Embedding 模型或分块策略不当，检索结果可能不准确。应对方案是使用中文优化的 bge-small-zh-v1.5 模型，调整 chunk_size（500）和 overlap（50）参数，设置相似度阈值（0.7）过滤低质量结果。可通过用户反馈数据持续优化检索策略。

流式响应兼容性风险。SSE（Server-Sent Events）在某些浏览器或代理环境下可能不兼容。应对方案是同时提供流式（SSE）和非流式两种问答接口，前端优先使用流式，降级时切换为非流式。

### 6.2 进度风险

知识图谱节点梳理工作量较大。如果两门课程的节点数量超过预期，可能导致第二阶段延期。应对方案是优先实现核心知识点（约 100 个），后续在开发过程中持续补充。可以先用核心节点完成全链路验证，再逐步丰富图谱内容。

Stable Diffusion 集成复杂度。SD 模型部署和 API 封装可能遇到环境配置问题。应对方案是准备降级方案：如果 SD 集成困难，改用静态教师图片+CSS 动画的简化方案，确保数字教师功能可用。SD 作为加分项而非必需项。

### 6.3 应急预案

如果项目整体进度落后超过一周，可按以下优先级裁剪功能：第一优先级保留知识图谱+RAG问答（核心功能），第二优先级保留数据分析仪表盘（展示效果），第三优先级保留用户认证（基础功能），第四优先级保留数字教师（加分项可简化），可裁剪文件上传问答和多轮对话记忆等高级功能。

## 七、开发规范

### 7.1 代码规范

Python 后端遵循 PEP 8 规范，使用 type hints 标注函数参数和返回值。API 路由函数使用 async/await 异步写法。统一异常处理，通过 FastAPI exception_handler 捕获所有未处理异常并返回统一错误格式。日志使用 Python logging 模块，记录请求参数、响应时间和错误堆栈。

前端使用 Vue 3 Composition API，CDN 方案通过全局变量引入 Vue 和 Element Plus。CSS 使用 CSS 变量定义主题色（--primary-color: #6366f1），确保配色统一。JavaScript 模块化封装，API 调用统一通过封装的 axios 实例。

### 7.2 Git 提交规范

提交信息格式：`<type>: <description>`，type 包括 feat（新功能）、fix（修复）、docs（文档）、refactor（重构）、test（测试）、chore（杂项）。分支管理：main 为主分支，develop 为开发分支，feature/* 为功能分支。

### 7.3 测试策略

后端使用 pytest 编写单元测试，覆盖 RAG 检索、图谱查询、认证逻辑等核心模块。API 接口使用 FastAPI TestClient 进行集成测试。前端通过手动测试验证各页面功能。在 M5 里程碑前完成全链路冒烟测试，确保核心流程无阻断问题。

## 八、答辩准备

### 8.1 演示流程

答辩演示按以下顺序进行：打开系统首页，展示仪表盘数据概览（KPI、趋势图、热门知识点）。进入知识图谱页面，展示两门课程的交互式知识图谱，点击节点查看详情，演示学习路径查询。进入智能问答页面，输入问题（如"什么是支持向量机"），展示流式回答过程，指向引用来源，演示多轮对话。进入数字教师页面，展示教师形象，进行文字交互，播放语音回答。进入数据分析页面，展示各维度图表。最后切换到 /docs 展示自动生成的 API 文档。

### 8.2 亮点提炼

知识图谱+RAG 双路检索是本项目最大亮点，不同于传统纯文档 RAG，本系统将知识图谱的结构化知识也融入检索增强，使回答更具逻辑性和系统性。8GB 显存下的模型交替加载策略也是工程亮点，解决了有限硬件条件下多模型共存的难题。数字教师模块的 Stable Diffusion+TTS 集成提升了系统的多模态交互体验。完整的前后端分离架构和规范的 API 文档体现了工程化开发能力。

### 8.3 答辩 PPT 大纲

PPT 共约 15 页，包括：项目背景与目标（2页）、系统架构设计（3页，含架构图和 RAG 管线图）、知识图谱设计与实现（2页）、RAG 问答核心实现（3页）、数字教师模块（2页）、数据分析可视化（1页）、项目总结与展望（2页）。
